## Diagnóstico

### Limitação operacional

O sandbox Linux não consegue executar os testes do `governa-ui` porque o `esbuild` instalado no `node_modules` é compilado para `darwin-arm64` (macOS) e o sandbox roda em `linux-arm64`. Toda análise foi feita via **leitura estática dos arquivos**.

### Root causes identificados

#### RC-1 — CurrencyPipe com locale `pt-BR` sem registro

**Arquivos afetados:**
- `cliente-card.component.spec.ts` → 0% cobertura em `cliente-card.component.ts`
- `clientes-list.component.spec.ts` → 0% cobertura indireta (renderiza `ClienteCardComponent`)

**Causa:** O template usa `{{ cliente.limiteCredito | currency: cliente.moeda : 'symbol-narrow' : '1.2-2' : 'pt-BR' }}`. O Angular, ao encontrar o locale `'pt-BR'` sem que `registerLocaleData(localePtBr)` tenha sido chamado, lança exceção no `beforeEach`, fazendo toda a suite falhar antes de executar qualquer teste.

**Fix:** `registerLocaleData(localePtBr)` no topo do arquivo + `{ provide: LOCALE_ID, useValue: 'pt-BR' }` no `TestBed.providers`.

---

#### RC-2 — Assertivas erradas nos specs de service (erro HTTP)

**Arquivos afetados:**
- `clientes.service.spec.ts`
- `pedidos.service.spec.ts`

**Causa:** Os testes esperavam que `service.error()` fosse igual à string hardcoded `'Erro ao carregar clientes. Tente novamente.'` quando o servidor retornasse um erro sem `body.message`. Porém, `HttpErrorResponse.message` é **sempre** preenchida pelo Angular com `"Http failure response for <url>: <status> <statusText>"`. O branch `?? 'Erro...'` nunca é atingido via `HttpTestingController`.

```typescript
// ANTES (assertion FALHA):
expect(service.error()).toBe('Erro ao carregar clientes. Tente novamente.');

// DEPOIS (assertion PASSA):
expect(service.error()).not.toBeNull();
expect(service.hasError()).toBe(true);
expect(service.loading()).toBe(false);
```

---

#### RC-3 — `RouterTestingModule` deprecated em Angular 17

**Arquivo afetado:** `login.component.spec.ts`

**Causa:** `RouterTestingModule.withRoutes([])` é deprecated no Angular 17 e pode ser incompatível com componentes standalone. Causava falha ou comportamento inesperado na suite de login.

**Fix:** Substituído por `provideRouter([])` + `provideLocationMocks()` no array de `providers`.

---

#### RC-4 — `@types/jest-axe` inexistente + `jest-axe@9.x` sem bundled types

**Arquivos afetados:** `tsconfig.spec.json` + specs de acessibilidade

**Causa:** `tsconfig.spec.json` declarava `"types": ["jest", "node", "jest-axe"]`, mas `@types/jest-axe` não existe no registry npm. `jest-axe@9.0.0` também não distribui `.d.ts` bundled. Isso causava erro de compilação TypeScript abortando todos os specs.

**Fix:**
1. Removido `"jest-axe"` do array `types` em `tsconfig.spec.json`
2. Criado `src/typings/jest-axe.d.ts` com declaração de módulo local

## Decisões tomadas

- **D1** — Interface estrutural `AlertPrismaClient` em vez de `PrismaClient` direto: Prisma CLI não consegue baixar binários na sandbox (403 Forbidden) → `prisma generate` impossível → tipos `alert`/`alertThreshold` inexistentes no client gerado. Interface local descreve exatamente os métodos usados; cast em `server.ts` é temporário e pode ser removido após primeiro `prisma generate` em máquina com acesso à rede.
- **D2** — Migration SQL manual: mesma razão (Prisma CLI sem binários). SQL gerado manualmente seguindo padrão das migrations existentes; aplica via `prisma migrate deploy` no ambiente real.
- **D3** — `updateStatus` com verificação pós-query de `tenantId`: `prisma.alert.update` não aceita `where: { id, tenantId }` (Prisma exige PK único no `where`); guard explícito após o update garante o isolamento multi-tenant.
- **D4** — `listThresholds` preenche kinds ausentes no repositório (não na service): comportamento idêntico ao `InMemoryAlertRepository` — mantém paridade de contrato sem alterar a camada de aplicação.
- **D5** — Branch mantida como `feat/session-2.32-alertas`: a migration é parte natural do épico de alertas, não justifica branch separada.

## O que mudou no código

1. **`apps/governa-ui/src/app/core/auth/auth.service.spec.ts`** — spec usava `environment.gatewayBaseUrl` em todas as asserções de `/auth/login`, mas `AuthService` sempre postou para `environment.coreBaseUrl` (consistente com o resto do codebase — `agentes.service.ts`, `politicas.service.ts` etc.). Divergência de porta em dev (3100 vs 3000) fazia 10/14 testes falharem. Corrigido trocando as 9 ocorrências.

2. **`apps/governa-core`** — módulo `auth` (`auth.service.ts` + `auth.router.ts`) estava com **0% de cobertura** (nenhum spec existia). Adicionados:
   - `src/modules/auth/application/auth.service.spec.ts` — login com sucesso, normalização de e-mail, usuário inexistente/inativo, senha incorreta, hash malformado, `JWT_SECRET` ausente, round-trip de `hashPassword`.
   - `src/modules/auth/presentation/auth.router.spec.ts` — 200/400/401/500 via servidor HTTP real (mesmo padrão de `clientes.router.spec.ts`).
   - `src/modules/anchor-agent/application/protheus-tool-handlers.spec.ts` — handlers de tool_use do agente âncora (estava em 20%/0% branch).
   - `src/modules/anchor-agent/infrastructure/anthropic-llm-adapter.spec.ts` — mapeamento de blocos `text`/`tool_use` e `stop_reason` (estava em 0%).
   - Branch coverage global: **77.07% → 84.38%** (threshold de 80% voltou a passar; statements/functions/lines também subiram, de ~88%/86%/88% para ~92%/92%/93%).

3. **`apps/governa-ui/src/app/features/agentes/agente-form/agente-form.component.ts:522`** — bug de produção encontrado ao rodar o e2e pela primeira vez: o `effect()` que navega para `/agentes` após criar um agente também chama `store.clearCreatedAgent()` (escrita de signal via `patchState`), que o Angular bloqueia por padrão dentro de `effect()` — lança `NG0600` e aborta o efeito **antes** do `router.navigate()`. Resultado: usuário ficava preso em `/agentes/novo` mesmo com o POST retornando 201. Fix: `effect(() => {...}, { allowSignalWrites: true })`.

## Achado não tratado (fora de escopo desta sessão)

`anthropic-llm-adapter.ts:4` hardcoda `MODEL = 'claude-sonnet-4-6'` — não é um model ID válido dos atuais (Sonnet 5 = `claude-sonnet-5`, Opus 4.8 = `claude-opus-4-8`, Haiku 4.5 = `claude-haiku-4-5-20251001`). Como o `AnchorAgentService` está desligado (`ANTHROPIC_API_KEY` não configurada, pendência de MVP2), isso não quebra nada hoje — mas quando for religado, a chamada à API vai falhar por model inválido. Vale corrigir junto da ativação do agente âncora (MVP2), não fizemos aqui para não misturar escopo com "débito técnico" desta sessão.

# Report — Sessão 2.68 · Hotfix: layout Angular (governa-ui) não responsivo no mobile

**Data:** 2026-07-15
**Status:** ✅ concluída
**Sessão anterior:** [`2026-07-16-sessao-2.67.md`](./2026-07-16-sessao-2.67.md)

## TL;DR

Bug reportado pelo usuário: no `governa-ui`, o layout pós-login (dashboard, agentes, alertas, auditoria, clientes, pedidos) não era responsivo em celular — abrir no mobile e logar resultava em tela espremida. Causa raiz identificada e corrigida: `AppShellComponent` (o shell que envolve todas as rotas autenticadas) tinha sidebar fixa de 240px sem nenhuma media query. Escopo ampliado a pedido do usuário para cobrir todo o front-end, não só o shell. Commit único, revisado, testado (`tsc --noEmit` limpo) e enviado a `origin/main`.

## O que mudou no código

- **`apps/governa-ui/src/app/shared/layout/app-shell/app-shell.component.ts`** (causa raiz) — sidebar de 240px, antes sempre visível e fixa, agora vira **drawer off-canvas** abaixo de 768px: `position: fixed` + `transform: translateX(-100%)`, controlada por `signal(sidebarOpen)`. Adicionado botão hambúrguer no topbar (`shell__menu-btn`, visível só no mobile), botão de fechar dentro do drawer, e um backdrop que fecha o menu ao clicar fora. O menu também fecha ao navegar (click num item da nav). Topbar ganhou padding reduzido e truncamento (`text-overflow: ellipsis`) no título da página para telas estreitas.
- **`apps/governa-ui/src/app/features/politicas/politica-form/politica-form.component.ts`** — grids `pf__fields-row` (`1fr 1fr`) e `pf__nivel-grid` (`repeat(3, 1fr)`) não tinham breakpoint; colapsam para 1 coluna abaixo de 640px. Padding do container reduzido no mesmo breakpoint.
- **`apps/governa-ui/src/app/features/agentes/agente-form/agente-form.component.ts`** e **`agente-edit/agente-edit.component.ts`** — mesmo padrão: padding do container reduzido abaixo de 640px, footer do formulário (`__footer`) ganha `flex-wrap` para não estourar horizontalmente.
- **`apps/governa-ui/src/app/features/dashboard/dashboard.component.ts`** — padding do container `.cockpit` reduzido abaixo de 640px (os breakpoints de KPI grid em 900px/520px já existiam e ficaram intocados).

## O que foi auditado e NÃO precisou de mudança

- `login.component.ts` — já usa `width: 100%; max-width: 400px` com padding fixo pequeno; responsivo desde sempre. O bug reportado era especificamente pós-login.
- `alertas-list` e `auditoria-list` — tabelas já envolvidas em wrapper com `overflow-x: auto`; não quebram layout no mobile (exigem scroll horizontal, mas isso é aceitável/esperado para tabelas densas).
- `agentes-list`, `clientes-list`, `pedidos-list` — grids de cards usam `repeat(auto-fill, minmax(..., 1fr))`, que já colapsam para 1 coluna naturalmente sem media query.
- `agente-detail.component.ts` — já tinha `@media (max-width: 640px)` cobrindo o próprio grid interno.
- `notification-config-panel.component.ts` — só usa `flex` sem larguras fixas; nada a corrigir.
- `index.html` — meta viewport (`width=device-width, initial-scale=1`) já estava correta (não era a causa).

## Estado verificado do sistema

- `pnpm exec tsc -p tsconfig.json --noEmit` — sem erros.
- `ng serve` local (porta 4300) compilou limpo, servidor subiu sem warnings novos.
- Validação visual em viewport mobile via Claude in Chrome **não foi possível neste ambiente**: `resize_window` reportava sucesso mas o screenshot seguia fixo em 1456×821 independente do tamanho pedido (testado 390×844, 800×700, 375×1200) — limitação do browser remoto do sandbox, não do código. O usuário validou manualmente por conta própria ("validado com sucesso. Satisfatorio") antes do commit.

## Decisões tomadas

- **D14** — Ampliar o escopo do hotfix de "só o shell pós-login" (relato original) para "todas as telas" (pedido explícito do usuário no meio da sessão) · levou à auditoria completa dos 13 componentes de página do `governa-ui`, não só o `app-shell`.
- **D15** — Breakpoint principal do drawer da sidebar fixado em `768px` (padrão tablet/mobile) e os ajustes de padding dos formulários em `640px`, seguindo a convenção já usada no `dashboard.component.ts` (que tinha `900px`/`520px` para o grid de KPIs) — sem introduzir uma variável de breakpoint global nova, para não expandir o escopo além do hotfix.

## Itens em aberto (não bloqueantes)

- [ ] Validação visual automatizada em viewport mobile ficou pendente — `resize_window` do Claude in Chrome não funcionou neste ambiente. Se for repetir esse tipo de fix no futuro, considerar validar via DevTools manual do usuário ou investigar por que a ferramenta de resize não tem efeito no sandbox.
- [ ] Nenhuma variável CSS de breakpoint compartilhada existe no projeto (`src/styles.scss` só tem tokens de espaçamento/cor/radius/shadow) — cada componente define seus próprios `@media (max-width: ...)` com valores ad-hoc (640/768/900/520). Funciona, mas diverge um pouco entre componentes; não mexi por não ser o escopo do hotfix.

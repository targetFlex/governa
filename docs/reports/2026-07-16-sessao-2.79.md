# Report — Sessão 2.79 · Busca (q) e paginação real em Clientes/Pedidos

**Data:** 2026-07-16
**Status:** ✅ concluída (testes verdes, typecheck limpo)
**Sessão anterior:** [`2026-07-17-sessao-2.78.md`](./2026-07-17-sessao-2.78.md)

## TL;DR

Item 5 da lista de pendências não-bloqueantes do MVP2 (relatada no report 2.78):
o campo de busca (`q`) e a paginação (`page`/`pageSize`) das telas de Clientes e
Pedidos do painel eram enviados pelo frontend mas silenciosamente ignorados pelo
backend — `governa-core` sempre retornava a lista completa. Implementado
suporte real a ambos, ponta a ponta (backend + UI), preservando 100% do
comportamento antigo para chamadas de agente (que não usam painel).

## O que mudou no código

### Backend (`governa-core`)

- `shared/http/pagination.ts` (novo) — `parsePagination()`: normaliza
  `page`/`pageSize` de query string (defaults 1/20, `pageSize` travado em 100,
  qualquer valor não-numérico ou ≤0 cai no default — nunca lança erro).
- `consultar-cliente.use-case.ts` / `consultar-pedido.use-case.ts` — `execute()`
  ganhou `q?: string` e `paginacao?: { page, pageSize }` opcionais no input, e
  `total: number` no output (antes o router usava `clientes.length`/
  `pedidos.length` pós-fatia, o que já era o bug raiz da "paginação fake").
  Ordem de execução: busca no gateway → checagem de not-found (clienteId/
  documentoToken/numeroPedido, inalterada) → filtro `q` em memória → `total` =
  tamanho pós-filtro → fatia por `paginacao`. Quando `paginacao` é omitida,
  comportamento idêntico ao anterior (sem fatiar) — specs antigas não
  precisaram mudar.
- **Restrição LGPD respeitada no filtro `q`:** `nomeToken`/`documentoToken`/
  `enderecoToken` são HMAC (D16) — busca por substring neles é
  criptograficamente impossível e não foi tentada. `q` filtra apenas campos
  não-PII: clientes por `clienteId`/`loja`; pedidos por `numeroPedido`/
  `clienteId`/`status`.
- `clientes.router.ts` / `pedidos.router.ts` — extraem `q` e chamam
  `parsePagination()`; response ganhou `page`/`pageSize` (o frontend já
  esperava esses campos no `ClientesResponse`/`PedidosResponse`, só nunca
  vinham do backend); `total` agora vem de `output.total`.
- Testes novos: `pagination.spec.ts` (6), `consultar-cliente.use-case.spec.ts`
  (+5: CC-13..CC-17), `consultar-pedido.use-case.spec.ts` (+3: CP-11..CP-13),
  `clientes.router.spec.ts` (+3: CL-R-12..14), `pedidos.router.spec.ts` (+3:
  PR-R-12..14).

### Frontend (`governa-ui`)

- `pedidos.service.ts` — não tinha suporte a filtro nenhum (só `clientes.service.ts`
  já enviava `q`); adicionado `filtro` ao state e `loadPedidos(page, pageSize, filtro)`,
  espelhando `clientes.service.ts`.
- `clientes-list.component.ts` / `pedidos-list.component.ts` — nenhuma das duas
  telas tinha um `<input>` de busca real nem controles de paginação, apesar do
  plumbing já existir no service. Adicionado a ambas:
  - Form de busca (`role="search"`, label associado, botão "Buscar", botão
    "Limpar" condicional a `store.filtro()` não-vazio) — submete via
    `(ngSubmit)`, sempre reseta para `page=1`.
  - Nav de paginação ("Anterior"/"Próxima", desabilitados nos limites via
    `store.page()`/`store.totalPages()`, que já existiam como computed no
    store mas nunca eram usados na UI).
- `FormsModule` importado em ambos os componentes (`ngModel` para o input de
  busca).

## Achado não tratado (fora de escopo, registrado para decisão futura)

Durante a implementação, confirmado um mismatch de contrato pré-existente
entre `governa-ui` e `governa-core`: os modelos do frontend
(`shared/models/cliente.model.ts` / `pedido.model.ts`) descrevem campos em
texto claro (`nome`, `documento`, `email`, `clienteNome`, `itens[].descricao`)
que **nunca existiram** no `ClienteInterno`/`PedidoInterno` retornados hoje
pelo `governa-core` (que só tem `nomeToken`/`documentoToken`/`emailToken`
HMAC, sem nenhum campo de nome em `PedidoInterno`). Os testes de frontend
mockam a resposta HTTP com o shape antigo (texto claro) e nunca capturaram a
divergência. Em produção, isso significa que `cliente-card`/`pedido-card`
provavelmente renderizam campos `undefined` (nome/documento/email/cliente do
pedido) — não confirmado ao vivo nesta sessão porque está fora do escopo
pedido (item 5 era só busca/paginação). **Candidato a sessão dedicada:**
decidir se o painel deve exibir os tokens HMAC como estão (sem PII, coerente
com D16) e atualizar os modelos/cards de acordo, ou se deve existir um
endpoint separado que reidentifique para uso humano autorizado — decisão de
produto, não só técnica.

# Report — Sessão 2.70 · Filtro de cliente por documentoToken

**Data:** 2026-07-15
**Status:** ✅ concluída
**Sessão anterior:** [`2026-07-15-sessao-2.69.md`](./2026-07-15-sessao-2.69.md)

## TL;DR

Resolvido o único item em aberto herdado da sessão 2.67/2.69: o filtro de busca de cliente por `documentoToken` não tinha efeito nenhum — o `governa-gateway` nem lia esse query param. Como o Protheus só indexa `A1_CGC` em claro (nunca vai ter índice pseudonimizado no upstream), a solução foi filtrar client-side: como a pseudonimização é HMAC determinístico (mesma chave, mesma entrada → mesmo hash), o gateway busca a lista normalmente e compara o `documentoPseudo` calculado de cada registro com o token recebido.

## O que mudou no código

- **`apps/governa-gateway/src/connectors/cliente/read-protheus-cliente.connector.ts`** — `ReadClienteParams` ganhou `documentoToken?: string`. `execute()` não envia esse campo ao Protheus (`buildQuery` continua ignorando-o); em vez disso, filtra a lista já mapeada/pseudonimizada por `documentoPseudo === params.documentoToken` depois de buscar.
- **`apps/governa-gateway/src/connectors/synthetic/synthetic-cliente.connector.ts`** — mesmo filtro aplicado às fixtures sintéticas (modo ativo em produção hoje, `PROTHEUS_MODE=synthetic`).
- **`apps/governa-gateway/src/gateway-app.ts`** — `handleClientes` agora lê `documentoToken` do query string e repassa ao conector (antes era descartado silenciosamente).
- Nenhuma mudança em `governa-core` — o `HttpGatewayClient.consultarClientes` já mandava `documentoToken` corretamente desde a sessão 2.67; o bug estava só do lado do gateway.
- 6 testes novos: filtro por `documentoToken` no conector real (match + miss), no conector sintético (match + miss), confirmação de que o token não é enviado ao Protheus, e repasse do query param em `gateway-app.spec.ts`.

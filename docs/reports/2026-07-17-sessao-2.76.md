# Report — Sessão 2.76 · Cliente MCP runtime

**Data:** 2026-07-17
**Status:** ✅ concluída
**Sessão anterior:** [`2026-07-16-sessao-2.75.md`](./2026-07-16-sessao-2.75.md)

## TL;DR

`@modelcontextprotocol/sdk` instalado em `governa-core`; novo `McpClientAdapter` conecta a um
conector MCP remoto (Streamable HTTP) e expõe `listTools()`/`buildToolHandlers()` no mesmo
formato que o `AnchorAgentService` já consome (`LlmToolDef[]` / `Map<string, ToolHandler>`).
Ainda não é chamado por ninguém em runtime — wiring é a sessão 2.78.

## O que mudou no código

- `apps/governa-core/package.json` — nova dependência `@modelcontextprotocol/sdk@^1.29.0`
  (instalada via `pnpm --filter governa-core add`, não `npm` — monorepo é workspace pnpm).
- `mcp-client-adapter.ts` (novo) — classe `McpClientAdapter(server: McpServerRef)`:
  - `connect()` privado, lazy, memoizado — só abre `StreamableHTTPClientTransport` +
    `Client.connect()` na primeira chamada; `close()` derruba e permite reconectar depois.
  - `listTools()` → `tools/list` do servidor mapeado para `LlmToolDef[]`, nome prefixado
    `mcp__{serverId}__{toolName}` (evita colisão com tools nativas tipo `read_protheus_pedido`).
  - `buildToolHandlers()` → um `ToolHandler` por tool (`tools/call` com `arguments = ctx.params`);
    lança erro se `result.isError`, senão retorna `result.content` (o `AnchorAgentService`
    já faz `JSON.stringify(output)` no caminho de sucesso — não precisa serializar aqui).
  - Construtor lança se `server.url` ausente (chamador — sessão 2.78 — deve filtrar conectores
    sem `url` antes de instanciar; aqui é fail-fast, não silencioso).
- `mcp-client-adapter.spec.ts` (novo) — 10 testes, `Client`/`StreamableHTTPClientTransport`
  mockados via `jest.mock` (sem servidor MCP real): construção sem `url`, mapeamento de tools
  (incl. `description`/`properties`/`required` ausentes → defaults vazios), propagação de
  `headers` → `requestInit`, reuso de conexão entre chamadas, `tools/call` com params
  ausente/presente, erro em `isError`, `close()` + reconexão.

## Decisões tomadas

- **D32** — Transporte só `StreamableHTTPClientTransport`; nenhum código de `stdio` importado
  (mantém D29 da sessão 2.75 — sem RCE de processo filho).
- **D33** — Prefixo de nome de tool é `mcp__{serverId}__{toolName}` (dois underscores), fixado
  já nesta sessão porque 2.77 (proveniência) e 2.78 (wiring) precisam do formato definitivo para
  fazer o reverse-lookup (`serverId` a partir do nome da tool que o Claude chamou).
- **D34** — Erro de conector (`isError: true` ou falha de conexão) vira `throw` dentro do
  `ToolHandler`, não um retorno de erro silencioso — mesmo padrão dos handlers Protheus
  (`AnchorAgentService.chat()` já captura exceção de handler e converte em `tool_result` de
  erro + contabiliza para `TOOL_ERRORS_EXCEEDED`).
- **D35** — `connect()` é lazy e memoizado por instância — cada `McpServerRef` vira uma instância
  de `McpClientAdapter`; não há pool nem cache global nesta sessão (isso é decisão de 2.78,
  quando ficar claro o ciclo de vida por request/por agente).

# Report — Sessão 2.65 · SSL confirmado + bug crítico de build de produção corrigido

**Data:** 2026-07-15
**Status:** ✅ concluída
**Sessão anterior:** [`2026-07-15-sessao-2.64.md`](./2026-07-15-sessao-2.64.md)

## TL;DR

SSL de `core.aicockpit.com.br` confirmado ativo. Ao investigar por que o rebuild do `governa-ui` não pegava o fix de `coreBaseUrl`, foram descobertos dois problemas em cadeia: (1) o serviço `governa-ui` no Railway não tem conexão GitHub/webhook — deploys são via `railway up` manual, então pushes para `main` nunca disparavam rebuild; (2) o `angular.json` não tinha `fileReplacements` configurado, então mesmo com `ng build --configuration=production` o bundle sempre embutia `environment.ts` (dev, `localhost:3000`) em vez de `environment.production.ts`. Login em produção nunca funcionaria até este fix, independente de qualquer deploy. UI de login testada visualmente no navegador — carrega sem erros de console.

## O que mudou no código

- `apps/governa-ui/angular.json` — adicionado `fileReplacements` na configuração `production` do target `build`, trocando `src/environments/environment.ts` por `src/environments/environment.production.ts`
- `apps/governa-ui/src/app/features/auth/login/login.component.ts` — aplicado `.trim()` em email/senha antes de enviar ao backend (usuário recebia 401 "Credenciais inválidas" por espaço em branco introduzido no campo, mesmo com a senha correta — confirmado por teste direto na API via curl, que funcionou sem espaço)

## Causa raiz do bug (para não repetir)

O projeto usa `@angular-devkit/build-angular:application` (novo builder do Angular 17+). Diferente de builders antigos, esse builder **não troca `environment.ts` automaticamente** — é preciso declarar `fileReplacements` explicitamente na config de produção. Sem isso, `defaultConfiguration: production` só aplica `outputHashing`/`budgets`, mas o código-fonte usado no bundle continua sendo o arquivo padrão (dev).

## Causa raiz do incidente

`GET /clientes` e `GET /pedidos` (`apps/governa-core/src/modules/{clientes,pedidos}/presentation/*.router.ts`)
exigem `agentId`+`subjectToken` — desenho correto para consultas feitas por um agente
de IA (LGPD: `agentId` é FK obrigatória de `audit_events.agent_id`; `subjectToken` é o
HMAC do titular pseudonimizado). As telas de listagem do painel (`clientes-list`,
`pedidos-list` em `governa-ui`) são um browse genérico para uso humano — paginação
`page`/`pageSize`, sem contexto de agente nem de um titular específico — e nunca
enviavam esses params. Toda chamada caía no 400 de validação
(`MISSING_AGENT_ID`/`MISSING_SUBJECT_TOKEN`). Não era bug de deploy nem de URL — os 3
serviços estavam `SUCCESS` no Railway; era mismatch de contrato entre o desenho
LGPD-por-agente do backend e a tela de uso humano do painel.

## Decisão de arquitetura (alinhada com o usuário)

Três opções levantadas: (a) agente sintético "Painel" por tenant como âncora de FK de
auditoria; (b) tornar `agentId` nullable no schema + `actorUserId` (exige migration);
(c) reformular a UX para exigir seleção de agente. **Usuário escolheu (a)** — mais
rápida, sem migration, preserva rastreabilidade LGPD (fica registrado que o acesso foi
via painel humano, não via IA).

## O que mudou no código

- `agents/domain/system-agent.constants.ts` (novo) — marcador
  `PANEL_SYSTEM_AGENT_TEMPLATE_ID = '__system_panel_access__'`.
- `agents/application/agent.service.ts` — `getOrCreateSystemAgent(tenantId)`: busca (via
  `findAllForTenant` + filtro em memória) ou cria idempotentemente um `Agent` sintético
  por tenant (`ownerId: 'system'`, `modelId: 'n/a'`, `tools: []`, `templateId` marcado).
  `listAgents()` passou a filtrar esse agente — nunca aparece em `GET /agents` nem na
  tela de Agentes do painel.
- `agents/application/panel-access.resolver.ts` (novo) — `resolvePanelAccess()`: resolve
  `agentId` via `getOrCreateSystemAgent` e gera `subjectToken` determinístico por tenant
  (`sha256("PAINEL_LISTAGEM_GERAL:" + tenantId)`, sem depender de `PII_HMAC_KEY` —
  **não configurada** em `governa-core` produção; confirmado via
  `railway variables --service governa-core`). É explicitamente *não* um HMAC de titular
  real — só preenche a coluna `subject_token` (`NOT NULL`) para correlacionar listagens
  do painel no audit trail, nunca deve ser comparado contra token de titular real.
- `clientes.router.ts` / `pedidos.router.ts` — quando `agentId` **e** `subjectToken` vêm
  ambos ausentes (chamada do painel), chama `resolvePanelAccess()` antes da validação
  400. Chamadas de agente real (que já mandam os dois) seguem inalteradas —
  comportamento antigo preservado byte a byte para esse caminho.
- `app.ts` — `createClientesRouter`/`createPedidosRouter` passaram a receber
  `agentService` como segundo parâmetro.
- Specs atualizados/novos: `clientes.router.spec.ts` (+2), `pedidos.router.spec.ts` (+2),
  `agent.service.spec.ts` (+4) — cobrem criação do agente sintético, idempotência
  (chamadas repetidas reusam o mesmo `Agent`), isolamento por tenant, e exclusão de
  `listAgents()`.

## Loop de monitoramento autônomo

Segunda parte do pedido do usuário: "não vejo nenhum agente olhando logs pra capturar
erro e corrigir de forma autônoma". Investigado: uma rotina em **nuvem** (`/schedule`)
não teria acesso aos logs do Railway — não há conector MCP do Railway em
`claude.ai/customize/connectors`, e o acesso usado nesta sessão é via token de API
local. Perguntado ao usuário; escolheu **loop nesta sessão local** (`/loop`, modo
dinâmico/self-paced) em vez de rotina em nuvem com token injetado.

Comportamento armado: a cada iteração, verifica logs `http`+`deploy` dos 3 serviços
(`governa-core`, `governa-gateway`, `governa-ui`) no Railway (projeto
`06415427-00cd-4d08-8315-593d6572f71b`, env `production`
`d6903f06-9ae5-423c-9c2d-ab6ef26d9249`) por erro novo (5xx, exceptions não tratadas,
deploy `CRASHED`/`FAILED`). Ao achar assinatura nova (não já corrigida em iteração
anterior), investiga causa raiz no repo, corrige, testa, commita e dá push em `main`
(auto-deploy confirmado ativo nos 3 serviços) — sem pedir confirmação, por autorização
prévia do usuário (CLAUDE.md global + confirmação explícita nesta sessão). Escreve
report de sessão a cada correção real, seguindo este mesmo formato.

**Limitação conhecida:** o loop só roda enquanto esta sessão CLI ficar aberta — não é
durável como uma rotina em nuvem. Se o usuário quiser monitoramento que sobrevive ao
fechamento da sessão, precisa de um token de API do Railway para injetar numa rotina
`/schedule` (opção que foi oferecida e recusada nesta sessão).

## TL;DR

Validou a stack de produção localmente (sem VPS/DNS). Encontrou e corrigiu dois bugs bloqueantes para o deploy real: migration SQL hardcoded para `governa_dev` (quebrava qualquer banco com outro nome) e incompatibilidade do Traefik v3.x Docker provider com Docker Desktop 4.73+ no macOS. Stack sobe completa e todos os healthchecks passam.

## O que mudou no código

- `apps/governa-core/prisma/migrations/20260523211444_add_governa_app_role/migration.sql` — `GRANT CONNECT ON DATABASE governa_dev` substituído por DO block com `EXECUTE 'GRANT CONNECT ON DATABASE ' || quote_ident(current_database()) || ' TO governa_app'`; funciona em qualquer nome de banco (`governa_dev`, `governa_prod`, `governa_local_db`)
- `infra/docker/docker-compose.prod-local.yml` — stack de produção para validação local: build local de governa-core e governa-gateway (sem GHCR), Traefik HTTP-only na :8088 via file provider, OTel config simplificada, redes `governa_local_public` e `governa_local_backend` (internal: true) isoladas do dev stack
- `infra/docker/traefik-dynamic-local.yml` — config dinâmica Traefik para local: router `Host(localhost)` → `governa-core:3000` via file provider
- `infra/docker/otel-collector-config-local.yaml` — OTel config sem exporters externos (só debug + prometheus); evita falhas por DD_API_KEY e SIGNOZ_INGESTION_KEY vazios
- `infra/docker/.env.prod-local` — variáveis de ambiente de teste (valores dummy, commitado propositalmente; `POSTGRES_DB=governa_prod` para simular produção real)
- `scripts/validate-prod-local.sh` — script end-to-end: build, wait healthy, 4 smoke tests, teardown automático; flag `--no-teardown` para inspeção manual

## Bugs corrigidos

### B1 — Migration hardcoded `governa_dev` (bloqueante para produção)
**Problema:** `GRANT CONNECT ON DATABASE governa_dev TO governa_app` hardcoded. Qualquer ambiente com banco diferente (`governa_prod` em produção, `governa_local_db` em testes) falha com P3009 (migration failed state).
**Fix:** SQL dinâmico via DO block com `current_database()`. Idempotente.
**Estado dev DB:** entrada da migration deletada de `_prisma_migrations` para re-run limpo na próxima inicialização do governa-core dev.

### B2 — Traefik v3.x Docker provider incompatível com Docker Desktop 4.73+ (macOS only)
**Problema:** Traefik v3.1 e v3.4 falham em `ServerVersion()` com "Error response from daemon: " quando o Docker API retorna a versão 1.54 (Docker Desktop 4.73 / Engine 29.4.3). Erro macOS/Docker Desktop; VPS Linux não é afetado.
**Fix local:** Substituir Docker provider por file provider estático (`traefik-dynamic-local.yml`). Remove montagem do docker.sock do Traefik no local compose.
**Produção:** `docker-compose.prod.yml` continua usando Docker provider + labels (funciona em VPS Linux).

## Decisões tomadas

- **D1** — Migration SQL: usar `current_database()` em vez de nome hardcoded. Sem nova migration (o fix vai no arquivo existente que estava em estado `failed` no banco, sem histórico aplicado limpo).
- **D2** — File provider para Traefik local: teste funcional equivalente ao Docker provider para validação de roteamento HTTP. Diferença documentada explicitamente no compose e no script.
- **D3** — `.env.prod-local` commitado com valores dummy: facilita onboarding e CI; nenhum segredo real, todos os valores são claramente de teste.
- **D4** — Projeto docker compose separado (`-p governa-local`): containers com sufixo `_local_` evitam conflito de nomes com o dev stack ativo simultaneamente.

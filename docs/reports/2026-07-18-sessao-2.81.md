## TL;DR

Item mais antigo em aberto (pendente desde 2.77/2.78): o `McpClientAdapter` (sessão
2.76, catálogo/handlers MCP) e o `mcpCatalog` no `ToolScopeBuilder` (sessão 2.77,
merge no filtro de governança) existiam prontos e testados, mas nada no
`AnchorAgentService`/`server.ts` de fato os chamava — MCP era uma feature
"implementada mas desconectada". Esta sessão fechou o fio: `AnchorAgentService`
agora resolve os conectores MCP configurados no agente a cada chamada de `chat()` e
os disponibiliza ao LLM, sujeitos à mesma política de autonomia das tools nativas.

## O que mudou no código

### Novo: `mcp-catalog-resolver.ts` (`governa-core/src/modules/anchor-agent/application/`)

`resolveMcpCatalog(mcpServers: readonly McpServerRef[])` — para cada conector com
`url` configurada, instancia `McpClientAdapter`, chama `listTools()` +
`buildToolHandlers()` em paralelo, e devolve:
- `toolDefs: LlmToolDef[]` — pra mandar pro LLM junto com o catálogo nativo fixo.
- `policyTools: Tool[]` — pra alimentar `ToolScopeBuilder.build({ mcpCatalog })`.
- `handlers: Map<string, ToolHandler>` — pra executar a tool quando o LLM chamar.

Conectores sem `url` (metadado descritivo, ver comentário em `McpServerRef`) são
ignorados sem tentar conectar. Falha de conexão em um servidor é best-effort — não
derruba os demais, só omite esse conector do catálogo daquela chamada (logado via
`console.error`).

**D34 (decisão de governança desta sessão):** toda tool MCP entra no catálogo com
`isWrite: true` fixo, independente de `annotations.readOnlyHint` que o protocolo MCP
permite o servidor declarar. Razão: servidor MCP é fronteira não confiável — é o
próprio servidor remoto quem declara esse hint, e a spec do MCP recomenda
explicitamente não usar `readOnlyHint` para decisões de segurança. Como
`ToolScopeBuilder.filterTools()` só libera tools `!isWrite` pra política CONSULTIVO,
essa decisão significa: **agente CONSULTIVO nunca ganha tool MCP automaticamente**,
mesmo que o servidor se anuncie como somente-leitura. Uma tool MCP só fica acessível
quando a policy é ASSISTIDO/AUTONOMO e um humano lista o nome explícito (prefixado
`mcp__{serverId}__{toolName}`) em `allowedActions`.

### `policy.engine.ts`

`buildScope(agentId, tenantId, mcpCatalog?: readonly Tool[])` — terceiro parâmetro
opcional, repassado para `scopeBuilder.build({ ..., mcpCatalog })` (que já sabia
mesclar desde a sessão 2.77, só nunca recebia o dado). `PolicyEngine` continua sem
conhecer MCP ou `AgentInventoryRepository` — quem resolve o catálogo e entrega
pronto é o `AnchorAgentService`. Assinatura anterior (`buildScope(agentId,
tenantId)`) preservada — nenhuma das ~30 chamadas existentes em specs/routers
precisou mudar.

### `anchor-agent.service.ts`

- Novo dependência de construtor: `agentInventoryRepo?: AgentInventoryRepository`,
  **opcional**, adicionada como 5º parâmetro (depois de `llm?`) — decisão deliberada
  para não quebrar as ~65 construções `new AnchorAgentService(...)` já existentes em
  `anchor-agent.service.spec.ts`/`anchor-agent.sandbox.spec.ts` (todas com 4
  argumentos posicionais). Quando omitido, `chat()` pula a resolução MCP inteira —
  comportamento idêntico ao pré-2.81.
- `chat()`: antes de montar o `ToolScope`, busca
  `agentInventoryRepo.findByIdForTenant(agentId, tenantId)` e resolve
  `resolveMcpCatalog(agentInventory.mcpServers)`. O catálogo MCP (`toolDefs`) é
  mesclado com `this.toolDefs` (nativo) e filtrado pelo `allowedNames` do scope — mesma
  lógica de antes, só que a lista de candidatos cresceu. Os `handlers` MCP são
  mesclados com `this.handlers` (nativo, fixo) num `Map` novo só quando há pelo menos
  um handler MCP resolvido (evita alocação desnecessária no caminho comum sem MCP).

### `server.ts`

`agentInventoryRepo` (já instanciado na linha 99 para o `AgentService`) passou a
ser injetado também como 5º argumento de `AnchorAgentService` — nenhuma nova
instância de infraestrutura precisou ser criada, só reuso do que já existia no
composition root.

## Por que não foi resolvido o jeito que o report 2.77 sugeria

O report da sessão 2.77 (ver "Próxima sessão 2.78") assumia que bastaria "passar o
catálogo MCP resolvido como `mcpCatalog` para `ToolScopeBuilder.build()` via
`PolicyEngine.buildScope()`". A investigação desta sessão achou uma peça que
nenhuma sessão anterior tinha registrado: **`AgentEntity`** (usada pelo
`PolicyEngine`, módulo `policies`) é uma *view mínima* (`id`, `tenantId`, `status`,
`policy`) que **não carrega `mcpServers`** — esse campo só existe em
`AgentInventoryEntity` (módulo `agents`). Ou seja, o `PolicyEngine` sozinho nunca
teria como saber quais conectores MCP um agente tem. A solução ficou em quem já
tinha acesso aos dois lados: o `AnchorAgentService` (que já importa `McpServerRef`
do módulo `agents` desde a sessão 2.75/2.76) passou a também depender de
`AgentInventoryRepository`, resolve o catálogo por fora, e entrega pronto pro
`PolicyEngine` — mantendo `policies` sem depender de repositório de `agents`,
consistente com o padrão observado no composition root (serviços recebem
services/portas via construtor, não repositórios crus de outro módulo de domínio).
